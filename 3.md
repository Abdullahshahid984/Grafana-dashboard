## What `volume_snapshot_enabled = true` actually backs up

It backs up **Persistent Volumes (PVs)** that are backed by **Azure Disk** (`azurerm_managed_disk`) only. Specifically, it uses the **Azure Disk CSI driver** to take snapshots — those snapshots land in your `snapshot_resource_group_name`.

It does **not** snapshot Azure Files shares or Blob-backed volumes — those are excluded from this mechanism entirely.

---

## The key question: do you actually have PVCs using Azure Disk?

To check, run this against each of your clusters:

```bash
# See all PVCs and their storage class
kubectl get pvc -A

# See storage classes and what provisioner they use
kubectl get storageclass
```

The provisioner to look for is `disk.csi.azure.com` — that's the Azure Disk CSI driver. If your PVCs are using that, then snapshots are relevant. If they're using `file.csi.azure.com` (Azure Files) or nothing at all (emptyDir, configMap, secret volumes), then this feature captures nothing meaningful.

Also check for actual PVs:

```bash
kubectl get pv -A
```

If this comes back empty or all statuses are `Released`/unused, you likely have no persistent disk volumes worth snapshotting.

---

## Likely scenarios for your setup

Given that you mentioned your applications are not heavy on volumes, there are a few common patterns:

| Scenario | What you'd see | Snapshot relevance |
|---|---|---|
| Stateless workloads only | No PVCs, or only emptyDir | None — safe to set `false` |
| Only Azure Files mounts | PVCs with `file.csi.azure.com` | Not captured by this |
| Azure Disk-backed PVCs | PVCs with `disk.csi.azure.com` | These get snapshotted |

---

## Recommendation

Run the `kubectl` commands above your cluster. If you find **no PVCs with `disk.csi.azure.com`**, then:

- `volume_snapshot_enabled = false` is the right setting
- It also means you avoid unnecessary snapshot resource group activity and potential snapshot costs
- The backup will still cover cluster-scoped resources (RBAC, CRDs, namespaced objects) since `cluster_scoped_resources_enabled = true` is separate

It's worth confirming in PRD specifically before changing it, since that's the tier where it would actually matter if data were lost.

## Premise

As part of the AKS backup and restore validation initiative, we needed a restore target cluster to validate that workloads can be successfully restored from the DEV cluster backup and brought back to a running state.

The original approach was to use a dedicated RST cluster (`aks-bfhaks-ihub-eus2-dev-rst-01`) in the DEV environment. However this caused multiple pipeline conflicts:
- Prometheus alert rule groups being restored by Azure Backup into the same resource group as DEV, conflicting with Terraform managed resources
- Workload Identity federated credential mismatches due to different OIDC issuer URLs between clusters

After evaluating all options, `aks-bfhaks-ihub-eus2-poc-02` was identified as the best restore target because:
- Already exists no new cluster provisioning needed
- Separate resource group (`rg-bfhaks-ihub-poc-eus2-dev-01`) completely isolated from DEV, zero Prometheus conflict
- Same subscription and networking restore from DEV backup is possible
- Created and owned by the platform team  full access for validation

## Why KEDA is needed

When DEV namespaces are restored to poc-02, the restored workloads include `ScaledObject` resources that depend on KEDA for autoscaling. Without KEDA installed on poc-02:
- HPA metrics cannot be fetched
- ScaledObjects cannot be reconciled
- Pods fail to scale correctly

KEDA must be installed on poc-02 before restore validation can proceed.

## PR Feedback 1 — Use a Data Source for the Snapshot RG

Instead of creating the resource group with `azurerm_resource_group`, reference it:

```hcl
data "azurerm_resource_group" "snapshot_rg" {
  name = local.conf.settings.snapshot_resource_group_name
}
```

Why this is the right pattern:
- The snapshot RG lifecycle is not owned by this module — it exists independently
- Terraform won't attempt to create or destroy it
- If someone deletes the RG externally, Terraform will error clearly rather than silently diverging

## PR Feedback 2 — Resource Lock on the Backup Vault

```hcl
resource "azurerm_management_lock" "backup_vault_lock" {
  for_each = azurerm_data_protection_backup_vault.backup_vault

  name       = "lock-${each.key}"
  scope      = each.value.id
  lock_level = "CanNotDelete"
  notes      = "Backup vault lock — deletion requires explicit lock removal."
}
```

Important behaviour to understand:

| Lock Level | Effect | Suitable Here |
|---|---|---|
| `CanNotDelete` | Blocks deletion, allows modifications and backup writes | ✅ Yes |
| `ReadOnly` | Blocks deletion and all writes | ❌ No — backup jobs will fail |

Additional considerations:
- `terraform destroy` will fail while the lock is active — you must remove the lock resource first or it will be blocked at the Azure API level
- You should also consider locking the **snapshot resource group** itself, since deleting it would orphan snapshot references inside the vault

```hcl
resource "azurerm_management_lock" "snapshot_rg_lock" {
  name       = "lock-snapshot-rg"
  scope      = data.azurerm_resource_group.snapshot_rg.id
  lock_level = "CanNotDelete"
  notes      = "Snapshot RG lock — contains AKS backup snapshots."
}
```

## What Actually Lands in the Snapshot Resource Group

Azure writes these automatically during each backup job — you do not manage them:

- **`Microsoft.Compute/snapshots`** — one snapshot per Azure Disk-backed PVC per backup run
- **Internal DataProtection metadata** — Azure-managed tracking objects, not visible to you directly

Your Backup Vault's managed identity must have **Disk Snapshot Contributor** role on this RG, otherwise backup jobs will fail with a permissions error.

---

## What `volume_snapshot_enabled = true` Actually Backs Up

This setting **only** captures Persistent Volumes backed by **Azure Disk** via the CSI driver. Nothing else.

| Volume Type | Provisioner | Snapshotted |
|---|---|---|
| Azure Disk | `disk.csi.azure.com` | Yes |
| Azure Files | `file.csi.azure.com` | No |
| Azure Blob | `blob.csi.azure.com` | No |
| emptyDir | In-memory / node-local | No |
| ConfigMap / Secret mounts | Not a PVC | No |

The rest of the backup — namespaces, deployments, RBAC, CRDs — is handled by `cluster_scoped_resources_enabled = true` and is completely independent of this setting.

## How to Check if This Feature Is Relevant to Your Clusters

Run these against each cluster, at minimum SIT and PRD:

```bash
# Check all PVCs across all namespaces
kubectl get pvc -A

# Check storage classes and their provisioners
kubectl get storageclass

# Check all PVs
kubectl get pv -A
```

What you are looking for:

```
NAME                PROVISIONER          RECLAIMPOLICY   ...
managed-csi         disk.csi.azure.com   Delete          # <-- this is what matters
azurefile-csi       file.csi.azure.com   Delete          # not captured
```

## Decision Matrix

| What you find | Action |
|---|---|
| No PVCs at all | Set `volume_snapshot_enabled = false` — feature irrelevant |
| PVCs but only `file.csi.azure.com` | Set `volume_snapshot_enabled = false` — nothing gets snapshotted anyway |
| PVCs with `disk.csi.azure.com` | Keep `true` — feature is active and meaningful |
| Mixed provisioners | Keep `true` but document that only disk-backed volumes are covered |

If you confirm no `disk.csi.azure.com` PVCs exist across your clusters, setting this to `false` is the clean and honest thing to do — it removes a setting that implies coverage you are not actually getting, avoids any snapshot RG activity, and eliminates a potential cost surface with no benefit.

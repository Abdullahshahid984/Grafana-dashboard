Wanted to share the full context and background on the AKS backup and restore validation work we have been doing over the past few weeks. This is a detailed writeup of everything we tried, what failed, and where we are now.

**Goal**

The goal of this work is to validate that AKS workloads can be successfully restored from the DEV cluster backup and brought back to a running state on a separate restore target cluster. This is part of our broader AKS backup and restore validation initiative to ensure our backup strategy is working correctly and workloads can recover in a disaster scenario.

**Attempt 1: Dedicated RST Cluster in DEV (aks-bfhaks-ihub-eus2-dev-rst-01)**

The first approach was to provision a dedicated RST cluster in the DEV environment using the bfhaksplatform pipeline and restore DEV backups to it. This seemed like the most straightforward approach but turned out to have several fundamental problems.

**Problem 1: Workload Identity (AADSTS700211)**

Each AKS cluster has a unique OIDC issuer URL. When workloads were restored to the RST cluster, pods got stuck in Init state because the Managed Identity federated credentials only covered the original DEV cluster's OIDC issuer. Entra ID rejected the token exchange requests with AADSTS700211: No matching federated identity record found, causing the CSI Secret Store to fail mounting Key Vault secrets, leaving all 71 namespace pods stuck in Init state.

To fix this we added a restore_clusters variable and workload_identity_restore resource to the workload onboarding Terraform code. The idea was to automatically create federated credentials for the RST cluster's OIDC issuer across all 71 workloads in one pipeline run. However when we tried to deploy via the workload onboarding pipeline it failed because:
- The excluded_cluster_names local value was resolving as known after apply at plan time, meaning Terraform could not evaluate it during the plan phase and the filter had no effect
- Multiple Terraform state conflicts occurred requiring manual terraform state rm operations directly on the HCP Terraform workspace via WSL
- The workspace state lock was triggered multiple times during repeated fix attempts causing further pipeline disruptions and blocking other team members

**Problem 2: Prometheus Alert Rule Group Conflict**

When the RST cluster was added to conf.yaml under the aks_cluster list, the app_alerts module in Phase 01 started creating Prometheus alert rule groups for every workload on the RST cluster. The rule groups are named after the workload (e.g. api-customerlookup-v1-AppAlerts) not the cluster, so when Azure Backup restored those same rule groups into the same DEV resource group (rg-bfhaks-ihub-eus2-dev-01) with the same name, Terraform tried to create them again and Azure rejected it with resource already exists error, breaking the DEV pipeline completely.

To fix this we added excluded_cluster_names logic to skip Prometheus rule group creation for the RST cluster. We also added exclude_prometheus_alerts: true flag to the RST cluster in conf.yaml and updated the schema file. However the conf.yaml changes in the upstream repo were not being picked up correctly by the workload onboarding pipeline at runtime because the pipeline was running on a cached older version of the code, causing the same conflict to repeat even after the fix was merged.

**Problem 3: Other Teams Affected**

Because the RST cluster was added to the shared DEV conf.yaml, other team members running the workload onboarding pipeline started hitting the same errors. Ashutosh's pipeline was failing because his pipeline build version was pointing to an older version of the code that still had RST cluster references. This caused confusion across the team and required additional communication and coordination to resolve.

After multiple PRs, manual state fixes, workspace lock issues, and pipeline disruptions across the team, the RST cluster approach was abandoned as too risky and too complex for the shared DEV environment.

**Attempt 2: POC-02 Cluster (aks-bfhaks-ihub-eus2-poc-02)**

After abandoning the RST cluster approach we evaluated all available options and identified aks-bfhaks-ihub-eus2-poc-02 as the best restore target for the following reasons:

| Reason | Detail |
|---|---|
| Already exists | No new cluster provisioning needed, saves time and avoids new infrastructure risk |
| Separate resource group | rg-bfhaks-ihub-poc-eus2-dev-01 completely isolated from DEV RG, zero Prometheus conflict |
| Same subscription | Azure Backup restore from DEV to POC-02 is possible within the same subscription |
| Platform team owned | Full access for restore validation, no permission blockers |
| No DEV pipeline impact | POC has its own bfhaksplatform pipeline and conf.yaml, DEV pipeline completely untouched |

**Known remaining challenge: Workload Identity**

POC-02 still has a different OIDC issuer URL from DEV. This means DEV workload Managed Identity federated credentials don't cover POC-02. Pods will still get stuck in Init state after restore due to AADSTS700211. This will be addressed in a follow-up PR by adding workload_identity_restore resource to the workload onboarding pipeline pointing to POC-02 cluster.

**Current Change: KEDA for POC-02**

When DEV namespaces are restored to POC-02, the restored workloads include ScaledObject resources that depend on KEDA for autoscaling. Without KEDA installed on POC-02:
- HPA metrics cannot be fetched from the external metrics API
- ScaledObjects cannot be reconciled by the KEDA operator
- Pods fail to scale and events show FailedGetExternalMetric and FailedCreate errors on ReplicaSets
- The Istio webhook also blocks pod creation since istiod-asm-1-26 service is not found

We have raised a PR to add KEDA for POC-02 using the existing POC KEDA identity (db28508a-a108-4120-8109-0dee64524a60) same client_id as POC-01 since both clusters are in the same POC environment.

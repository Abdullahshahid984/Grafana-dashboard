## Premise

### Goal
The goal of this work is to validate that AKS workloads can be successfully restored from the DEV cluster backup and brought back to a running state on a separate restore target cluster.

### Attempt 1: Dedicated RST Cluster in DEV (`aks-bfhaks-ihub-eus2-dev-rst-01`)

The first approach was to create a dedicated RST cluster in the DEV environment and restore DEV backups to it.

**Problem 1: Workload Identity (AADSTS700211)**

Each AKS cluster has a unique OIDC issuer URL. When workloads were restored to the RST cluster, pods got stuck in Init state because the Managed Identity federated credentials only covered the original DEV cluster's OIDC issuer. Entra ID rejected the token exchange requests with `AADSTS700211: No matching federated identity record found`, causing the CSI Secret Store to fail mounting Key Vault secrets.

To fix this we added `restore_clusters` variable and `workload_identity_restore` resource to the workload onboarding Terraform code to automatically create federated credentials for the RST cluster across all 71 workloads. When deployed via the workload onboarding pipeline it failed because:
- The `excluded_cluster_names` local was resolving as `known after apply` at plan time so the filter had no effect
- Multiple Terraform state conflicts occurred requiring manual `terraform state rm` operations on the HCP Terraform workspace
- The workspace state lock was triggered multiple times during repeated fix attempts causing further pipeline disruptions

**Problem 2: Prometheus Alert Rule Group Conflict**

When RST cluster was added to `conf.yaml` under `aks_cluster`, the `app_alerts` module in Phase 01 started creating Prometheus alert rule groups for every workload on the RST cluster. Azure Backup had already restored those same rule groups into the same DEV resource group (`rg-bfhaks-ihub-eus2-dev-01`) with the same name. When Terraform tried to create them Azure rejected it with `resource already exists` error, breaking the DEV pipeline.

To fix this we added `excluded_cluster_names` logic to skip Prometheus rule group creation for the RST cluster. However the `conf.yaml` changes in the upstream repo were not being picked up correctly by the workload onboarding pipeline at runtime, causing the same conflict repeatedly.

**Problem 3: Pipeline Breaking for Other Teams**

Because RST cluster was added to the shared DEV `conf.yaml`, other team members running the workload onboarding pipeline started hitting the same errors. Ashutosh's pipeline was failing because his build was pointing to an older version of the code that still had RST cluster references.

After multiple PRs, state fixes, and pipeline disruptions, the RST cluster approach was abandoned as too risky for the shared DEV environment.

### Attempt 2: POC-02 Cluster (`aks-bfhaks-ihub-eus2-poc-02`)

After abandoning the RST cluster approach, we evaluated using the existing `aks-bfhaks-ihub-eus2-poc-02` cluster as the restore target.

**Why POC-02 is the right choice:**

| Reason | Detail |
|---|---|
| Already exists | No new cluster provisioning needed |
| Separate resource group | `rg-bfhaks-ihub-poc-eus2-dev-01` completely isolated from DEV RG, zero Prometheus conflict |
| Same subscription | Azure Backup restore from DEV to POC-02 is possible |
| Platform team owned | Full access for restore validation |
| No DEV pipeline impact | POC has its own pipeline and conf.yaml, DEV pipeline untouched |

**Why POC-02 needs KEDA:**

When DEV namespaces are restored to POC-02, the restored workloads include `ScaledObject` resources that depend on KEDA for autoscaling. Without KEDA installed on POC-02:
- HPA metrics cannot be fetched from the external metrics API
- ScaledObjects cannot be reconciled by the KEDA operator
- Pods fail to scale and events show `FailedGetExternalMetric` errors
- Restore validation cannot be completed

KEDA must be installed on POC-02 before restore validation can proceed.

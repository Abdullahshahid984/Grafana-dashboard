AKS Backup Terraform — PR Changes Summary

Below is a summary of the changes made to the AKS backup Terraform configuration and the reasoning behind each decision.

─────────────────────────────────────
Changes Made
─────────────────────────────────────

1. Resource Lock on Backup Vault
Added an azurerm_management_lock resource with CanNotDelete on the backup vault. This prevents accidental deletion of the vault and all backup data it holds. Any future deletion attempt will require the lock to be explicitly removed first.

2. Data Source for Snapshot Resource Group
Replaced the previously hardcoded resource group string references with a proper data source block (data.azurerm_resource_group.snapshot_rg). This means Terraform now holds a real reference to the existing RG rather than constructing its ID and name through string interpolation. If the RG does not exist at plan time, Terraform will fail early and clearly rather than silently diverging.

3. Removed Unused RBAC Role Assignments
Removed the following two role assignments from backup.tf as they are no longer exercised:
  - vault_snapshot_contributor (Disk Snapshot Contributor on snapshot RG)
  - vault_data_operator (Data Operator for Managed Disks on snapshot RG)
These were only relevant when volume_snapshot_enabled = true. Since that setting has been changed to false (see investigation below), these role assignments serve no purpose and have been removed to keep the configuration clean.

─────────────────────────────────────
What AKS Backup Covers
─────────────────────────────────────

AKS backup is made up of two independent parts:

  1. Cluster state (Kubernetes manifests) — stored in the storage account blob container. This includes deployments, StatefulSets, DaemonSets, ConfigMaps, Secrets, Helm-managed resources, RBAC definitions, CRDs, namespace definitions, and all other Kubernetes API objects. This is controlled by cluster_scoped_resources_enabled = true and is unaffected by any of the changes above.

  2. Persistent Volume data — snapshotted as Microsoft.Compute/snapshots into the snapshot resource group. This is controlled by volume_snapshot_enabled and only applies to PVCs backed by Azure Disk (disk.csi.azure.com provisioner).

─────────────────────────────────────
Investigation: volume_snapshot_enabled
─────────────────────────────────────

To determine whether volume snapshots are relevant to our workloads, I ran kubectl get pvc -A and kubectl get storageclass across all IHUB environments.

Findings:
  - Only one PVC exists across all IHUB clusters
  - Workload:      cost-analysis-agent-victoria-metrics
  - Namespace:     kube-system
  - Size:          8Gi
  - Storage Class: default (provisioner: disk.csi.azure.com)

However, according to Microsoft documentation, the following namespaces are excluded from AKS backup scope by default and cannot be backed up:
  - kube-system
  - kube-node-lease
  - kube-public

Since the only PVC present resides in kube-system, it falls outside the backup scope entirely. volume_snapshot_enabled = true was therefore providing no coverage.

Decision: changed volume_snapshot_enabled to false.

─────────────────────────────────────
Snapshot Resource Group — Why It Stays
─────────────────────────────────────

The snapshot_resource_group_name argument is a mandatory field in the Terraform provider for azurerm_data_protection_backup_instance_kubernetes_cluster. It cannot be removed even when volume_snapshot_enabled = false — the provider will throw a validation error if it is absent.

Since no snapshots will be written to this RG, creating a dedicated separate resource group for it would add unnecessary overhead. The decision was made to reference the existing platform resource group via the data source. This keeps the configuration clean without introducing a resource group that will remain permanently empty.

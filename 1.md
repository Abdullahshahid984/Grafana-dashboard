## Summary

This PR adds support for excluding specific AKS clusters from Prometheus alert rule creation in the workload onboarding pipeline. This is needed to prevent pipeline failures when a restore/validation cluster is added to `conf.yaml`.

---

## Premise

During AKS backup restore validation, the RST cluster `aks-bfhaks-ihub-eus2-dev-rst-01` was added to `conf.yaml` under `aks_cluster`. This caused the `app_alerts` module in Phase 01 to create Prometheus alert rule groups for every workload on the RST cluster. However, Azure Backup had already restored those same rule groups into the same resource group with the same name — so when Terraform tried to create them, Azure rejected it with a **"resource already exists"** error, breaking the DEV pipeline.

---

## Changes

### `phases/01-per-instance/locals.tf`
- Added `excluded_cluster_names` local  automatically derived from `conf.yaml` by reading clusters with `exclude_prometheus_alerts: true`
- Fully driven by conf.yaml

### `phases/01-per-instance/main.tf`
- Added `excluded_cluster_names = local.excluded_cluster_names` inside the existing `module "app_alerts"` block

### `modules/app_alerts/variables.tf`
- Added `excluded_cluster_names` variable  accepts a list of cluster names to skip
- Defaults to `[]`  zero impact on existing behavior when not in use

### `modules/app_alerts/main.tf`
- Added `if !contains(var.excluded_cluster_names, cluster.name)` filter inside the inner for loop of the `azurerm_monitor_alert_prometheus_rule_group` resource

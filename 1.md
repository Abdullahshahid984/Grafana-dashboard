Hi , I investigated further and Follow up on the kanasmc.alldata.net 401 issue (case 20258796) traced this to root cause via direct testing in APIM + AKS logs. Sharing findings so this can get fixed properly.

**Primary issue 401 from kanasmc.alldata.net**
AKS pod logs show the actual outbound call and failure:
`CustomHttpClientErrorException: 401 Unauthorized on GET request for https://kanasmc.alldata.net:443/caseretriever/case/20258796`
This fires from `KanaCaseValidationController` → `KanaCaseValidationService.validateKanaCase()`. The credential/token being sent to Kana's case retrieval service is being rejected invalid, expired, or missing.

**Key comparison test:** I ran the same case ID (20258796) through both parallel APIM APIs:
- `CIMS API` (backend: App Service `appsvc-*-ebo-ei-ci...`) → 200 OK, "KANA case validated successfully"
- `CIMS API AKS` (backend: AKS, created 7/20) → 500 Internal Server Error

Same case ID, same downstream target, different result. This suggests AKS may be using a different (possibly stale) `KANA_API_AUTHORIZATION_KEY` than whatever credential the App Service path uses successfully. Worth comparing the two directly AKS's version was newly added via Key Vault secret in the 7/22 deployment (revision 3, chart harness-87).

**Secondary issue exception handler bug (AKS-specific)**
When the 401 comes back, AKS's own exception handler crashes trying to process it, which is why you see a 500 instead of a clean 401:
`GlobalControllerExceptionHandler.handleHttpClientErrorException` → `IllegalStateException: Could not resolve parameter [0] in KanaErrorMessage No suitable resolver`

Likely a parameter binding issue in the `@ExceptionHandler` method signature the `HttpClientErrorException` parameter doesn't seem to be resolving correctly. This is masking the real error and should be fixed regardless of the credential issue, so future failures surface a clean 401 instead of an opaque 500.

**Suggested next steps:**
1. Compare/verify the `KANA_API_AUTHORIZATION_KEY` secret used by AKS against whatever credential the App Service uses for the same kanasmc.alldata.net calls
2. Check the secret's expiry in Key Vault
3. Fix the exception handler's parameter resolution so 401s from Kana surface cleanly instead of 500ing

Happy to share the full stack traces/screenshots if useful. This is confirmed not to be an AKS networking, DNS, VirtualService, or Istio issue pod logs, Envoy routing table, and VirtualService config all check out clean; this is a credential/application-level issue specific to the Kana integration code.

Investigated the volume_snapshot_enabled = true setting and whether it is relevant to our AKS usage. Below is the full picture of what the AKS backup covers.

AKS backup has two independent parts:

  1. Kubernetes resource manifests — stored in the storage account (sasbfhaksihubpocbackup) under blob container (blobakspocbackup). This covers:
     - Deployment specs, ReplicaSets, DaemonSets
     - ConfigMaps and Secrets
     - RBAC definitions (Roles, RoleBindings, ClusterRoles)
     - CRDs and custom resources
     - Namespace definitions
     - Service accounts and service definitions
     - Any other Kubernetes API objects present at backup time
     This is the cluster state backup — everything needed to reconstruct what was running. Controlled by cluster_scoped_resources_enabled = true.

  2. Persistent Volume data — snapshotted as Microsoft.Compute/snapshots into the snapshot resource group. Controlled by volume_snapshot_enabled = true.

These two parts are completely independent. The storage account is always required as long as the backup extension is installed, regardless of the volume snapshot setting.

Regarding volume_snapshot_enabled = true specifically — this setting only snapshots Persistent Volumes backed by Azure Disk (disk.csi.azure.com provisioner). It does not capture Azure Files, Blob-backed volumes, emptyDir, or ConfigMap/Secret mounts.

To determine if we have any qualifying volumes, I ran kubectl get pvc -A and kubectl get storageclass against the cluster. The output shows one active PVC:

  Namespace:     kube-system
  Workload:      cost-analysis-agent-victoria-metrics
  Size:          8Gi
  Access Mode:   RWO
  Storage Class: default (provisioner: disk.csi.azure.com)

The default storage class uses disk.csi.azure.com, which means this PVC is Azure Disk-backed and will be snapshotted on every backup run.

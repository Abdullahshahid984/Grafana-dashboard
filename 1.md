Hi team,

Following up on the question about why the lineage data needs to be reloaded after pod restarts I did a full investigation inside the cluster.

---

**1. Where uploaded files actually go**

I exec'd into the `collibra-edge-session-manager` pod and traced the upload path:

```
kubectl exec -it collibra-edge-session-manager-8455c44ddb-64nsq \
  -n app-collibraedge-v1 -- /bin/bash
```

```
ls -R /tmp/session-workspaces/

/tmp/session-workspaces/:
drivers

/tmp/session-workspaces/drivers:
VmD1imnk6-so2BFLCqlzJrxHGvzzKJpLTzvdy15CSOY
```

The uploaded lineage/driver file lands specifically at:
```
/tmp/session-workspaces/drivers/<filename>
```

---

**2. What backs that storage**

From `kubectl describe pod collibra-edge-session-manager` on the same pod, the Volumes section shows:

```
Volumes:
  tmp                      → Type: EmptyDir  
```

`EmptyDir` means the storage is created when the pod starts and permanently deleted when the pod stops or restarts. There is no `PersistentVolumeClaim` present.

---

**3. Are any PVCs being used**

I checked for PVCs and PVs in the namespace:

```
kubectl get pvc -n app-collibraedge-v1
→ No resources found in app-collibraedge-v1 namespace
```

The PVs that exist in the cluster are bound to `kube-system` and `app-collibradataquality-v1` — completely separate namespaces. The `app-collibraedge-v1` namespace has zero persistent storage of any kind.

---

**4. Is this path shared across all pods**

I also exec'd into `collibra-edge-cd` and confirmed:

```
/tmp  → contains only "helm" (no session-workspaces)
/data → does not exist
```

**Root Cause Summary**

| Layer | Finding |
|-------|---------|
| Where file lands | `/tmp/session-workspaces/drivers/` in `session-manager` pod |
| What backs it | `emptyDir` — ephemeral, pod-local storage |
| Any PVC in namespace | None — zero persistent storage exists in `app-collibraedge-v1` |

The lineage data is lost on  pods restart because the upload path is backed by ephemeral storage with no persistence layer behind it.

---

**Recommended Path Forward**

For the short term we will continue to manually reload the file after pod restarts as we do now.

For a long term fix there are two options:

- **Option A — Mount a PVC:** Modify the `session-manager` deployment to mount a new PVC at `/tmp/session-workspaces`. The file would survive pod restarts.
- **Option B — Azure Blob/Files (preferred for production):** Back the path with Azure storage so any new pod automatically has access to the file without any manual reload.

Happy to work through either option with the team when ready.

## Summary

Refactored AKS backup configuration to use the existing platform resource group for snapshot storage instead of creating a dedicated snapshot resource group.

## Changes

**File Modified:** `modules/main/backup.tf`

- Removed `azurerm_resource_group.snapshot_rg` resource block — no longer creating a separate resource group for AKS snapshots.
- Removed `snapshot_rg_name` field from the `backup_instances/ihub-poc-eus2/conf.yaml` local — no longer required in configuration.
- Removed `snapshot_rg_block` field from the `schema/instance-conf.yaml`  no longer required in configuration.
- Updated `azurerm_data_protection_backup_instance_kubernetes_cluster.aks_backup_instance` to set `snapshot_resource_group_name` to `local.conf.settings.resource_group_name`.
- Updated RBAC role assignments `vault_snapshot_contributor` and `vault_data_operator` to scope against the existing platform resource group instead of the removed snapshot resource group resource.

## Reason

The service principal used during Terraform execution does not have `Microsoft.Resources/resourceGroups/read` at subscription scope, which caused an `AuthorizationFailed` (HTTP 403) error when Terraform attempted to create and validate the snapshot resource group. Since the existing platform resource group already hosts all related AKS backup resources (storage account, backup vault, etc.), creating a separate snapshot resource group provides no additional value and introduces an unnecessary permission boundary.
